## 🏗 1. Описание системы и Стек
**Цель:** Telegram-бот для интервального заучивания стихотворений (на русском и английском) с автоматической проверкой чтения голосом.

**Бэкенд:**
- **Фреймворк:** FastAPI (Python 3.11+)
- **База данных:** PostgreSQL 16
- **ORM:** SQLAlchemy 2.0 (Async) + asyncpg
- **Векторный поиск:** pgvector (планируется для рекомендаций)
- **ML / NLP:** Vosk (оффлайн распознавание речи, Voice-to-Text), difflib (сравнение текста), `sentence-transformers` (эмбеддинги).

**Фронтенд (Бот):**
- **Фреймворк:** aiogram 3.x
- **Клиент API:** `httpx` (асинхронные запросы к FastAPI бэкенду)
- **Стейт-менеджмент:** встроенная в aiogram FSM (Finite State Machine) для процесса записи голоса.

**Инфраструктура:** Docker Compose (`db`, `backend`, `bot`).

## 🚫 2. КРИТИЧЕСКИЕ ПРАВИЛА (Никогда не нарушай)

Проект падал из-за мелких недочетов. Явные архитектурные конвенции:

### 2.1. Единый `Base` (SQLAlchemy)
**ЗАПРЕЩЕНО:** Определять `class Base(DeclarativeBase): ...` внутри отдельного файла модели (например, `user.py` или `poem.py`). Это приводит к циклическим импортам и перезаписи метаданных таблиц.
**ПРАВИЛО:** Модели всегда импортируют `Base` только из `app.database`:
```python
from app.database import Base
class User(Base): ...
```
В файле `backend/app/models/__init__.py` прописаны и импортированы все модели, чтобы `Base.metadata.create_all` подхватывал все таблицы из одного места (`main.py`).

### 2.2. Роутинг и Префиксы в FastAPI
**ЗАПРЕЩЕНО:** Дублировать префиксы (например, писать `safe_include(..., prefix="/poems")` в роутере, если внутри файла `poems.py` уже стоит `router = APIRouter(prefix="/poems")`). Раньше это приводило к 404 на `/api/v1/poems/poems/`.
**ПРАВИЛО:** 
1. В `main.py` роутер подключается ровно с одним глобальным префиксом: `app.include_router(router, prefix="/api/v1")`.
2. В файле-агрегаторе `api/router.py` мы инклудим под-роуты БЕЗ префикса: `safe_include("poems")`.

### 2.3. Синхронизация Pydantic схем и SQLAlchemy моделей
**ЗАПРЕЩЕНО:** Добавлять поля в `PoemResponse` (например `themes: list[str]`), если в БД в модели `Poem` нет реальной колонки `themes`. Иначе SQLAlchemy вернет ошибку Missing Attribute.
**ПРАВИЛО:** Если нужно новое поле, сначала пропиши его в БД (`backend/app/models/...`), и только потом в схему `backend/app/schemas/...`.

### 2.4. Методы SQLAlchemy 2.0
**ЗАПРЕЩЕНО:** Использовать старые методы типа `not_in()` или синхронные вызовы.
**ПРАВИЛО:** Использовать только стиль async 2.0. Вызов отрицания — `.notin_()`. Запросы всегда выполняются через `await session.execute(select(...))`.

## 🗺 3. Roadmap (Планы на будущее)
**Текущее состояние:** Есть MVP. Работает регистрация пользователя, алгоритм SM-2 (spaced repetition), и голосовая проверка стихов через Vosk STT.

**Для чего нужен ИИ (ты):** Пользователь будет просить тебя реализовать следующие фичи. Действуй исходя из этих планов:

1. **Геймификация и Механики:**
   * **Система опыта:** Добавить в модель `User` поля для streak (серия дней), XP (опыт), level (уровень).
   * **Построчный режим:** Режим "stanza-by-stanza", где бот выдает стих не целиком, а кусками (строфами), чтобы учить было легче.
   * Ачивки и Leaderboard.
2. **Умные рекомендации:**
   * Реализация генерации векторов (`embedding`) для стихов при инициализации БД (папка `seed`).
   * Диалоговый опросник (preference elicitation): бот задает пару вопросов (настроение, эпоха), а FastAPI через `pgvector` и `sentence-transformers` подбирает идеальный стих по косинусному расстоянию.
3. **Парсинг:**
   * Написать `parser.py` (используя `aiohttp` и `BeautifulSoup4`), чтобы бот по команде мог сам находить текст стихотворения на сайтах (poemhunter.com, stihi.ru) и добавлять в БД.
4. **Миграции БД (ОЧЕНЬ ВАЖНО):**
   * Прямо сейчас БД создается так: `await conn.run_sync(Base.metadata.create_all)`.
   * **Задача на будущее:** Внедрить `Alembic`, чтобы структура БД (поля, таблицы) могла обновляться без потери данных (drop table).

## 📂 4. Структура проекта

Структура разделена на две части (каждая - свой Docker-контейнер).

### Backend (`/backend/app/`)
* `main.py`: Точка входа, старт `FastAPI`, создание таблиц (до появления alembic).
* `database.py`: Конфигурация PostgreSQL, `engine`, `async_session`, класс `Base`.
* `config.py`: Настройки из `.env` через `Pydantic-Settings`.
* `api/`: Роуты FastAPI (`poems.py`, `users.py`, `memorization.py`, `recommendations.py`, агрегатор `router.py`).
* `models/`: Схемы таблиц базы данных (`poem.py`, `user.py`, `memorization.py`). 
* `schemas/`: Pydantic модели проверки входящих и исходящих JSON.
* `services/`: Бизнес-логика, не зависящая от HTTP (`voice_evaluator.py`, `spaced_rep.py`).
* `seed/`: Заполнение базы дефолтными стихами при первом запуске (`seed_poems.py`).

### Bot (`/bot/app/`)
* `main.py`: Инициализация `aiogram.Bot` и `Dispatcher`.
* `config.py`: Токен и URL бэкенда.
* `handlers/`: Логика обработки сообщений от юзера (`start.py` для текстовых команд, `voice.py` для приема голосовых через FSM).
* `services/api_client.py`: Обёртка над `httpx` для похода в `backend:8000`.
